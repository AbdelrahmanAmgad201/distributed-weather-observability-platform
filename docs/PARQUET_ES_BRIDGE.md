# Parquet to Elasticsearch Bridge

## Overview

The `parquet-es-bridge` is a lightweight Python service responsible for bridging the gap between the historical Parquet files generated by the Central Station and the Elasticsearch analytics engine. 

It acts as an automated ETL (Extract, Transform, Load) pipeline, ensuring that all historical weather data is continuously available in Kibana for visualization.

---

## Why a separate bridge service?

The Central Station is optimized for high-throughput stream processing and fast key-value lookups (Bitcask). While it batches and archives historical data to Parquet files (a highly compressed columnar format), directly writing to Elasticsearch from the Java application would couple the streaming logic too tightly to the analytics layer and slow down stream processing.

By decoupling this into a separate service:
1. **Separation of Concerns:** The Java app only worries about Parquet I/O.
2. **Reliability:** If Elasticsearch goes down, the bridge can retry without affecting the main telemetry pipeline.
3. **Efficiency:** The bridge uses bulk indexing to efficiently push large Parquet batches into Elasticsearch.

---

## Architecture & Logic

```mermaid
flowchart LR
    CS[Central Station] -->|Writes| PARQUET[(Parquet Files)]
    PARQUET -->|watchdog / polling| BRIDGE[Parquet-ES Bridge]
    BRIDGE -->|pyarrow| MEMORY[Memory Dataframes]
    MEMORY -->|elasticsearch-py bulk()| ES[(Elasticsearch)]
```

### 1. File Monitoring (`watchdog`)
The bridge monitors the shared `/data/parquet` directory. It uses a polling observer (`watchdog`) to detect when a new `.parquet` file is fully written and closed. 

### 2. State Management (Processed Files)
To prevent duplicate data in Elasticsearch, the bridge maintains an in-memory set of `processed_files`. When a file is successfully indexed, its filename is recorded. (In a production environment, this state could be backed by Redis or Bitcask).

### 3. Data Transformation (`pyarrow`)
When a new file is detected, it is read using Apache Arrow (`pyarrow`). The script converts the columnar Parquet data into a list of Python dictionaries.

Since the Java application serializes the data using Avro-reflection, nested objects (like `weather.temperature`) are mapped perfectly into Elasticsearch JSON documents.

### 4. Bulk Indexing
The records are packaged into Elasticsearch bulk actions and pushed to the `weather-telemetry` index. If the index does not exist, Elasticsearch creates it automatically.

## Deployment Configuration

The bridge relies on shared volumes to access the Parquet data.

### Docker Compose
It mounts the `parquet_data` named volume:
```yaml
parquet-es-bridge:
  build: ./parquet-es-bridge
  volumes:
    - parquet_data:/data/parquet:ro
```

### Kubernetes (Minikube)
It shares a `ReadWriteOnce` PersistentVolumeClaim (`parquet-pvc`) with the `central-station` pod. Since Minikube runs on a single node, both pods can mount the same PVC simultaneously without issue.

### Environment Variables

| Variable | Description | Default |
|----------|-------------|---------|
| `PARQUET_DIR` | Path to the directory containing Parquet files. | `/data/parquet` |
| `ES_HOST` | Elasticsearch URL. | `http://elasticsearch:9200` |
| `ES_INDEX` | Target index in Elasticsearch. | `weather-telemetry` |
| `POLL_INTERVAL` | Polling interval in seconds. | `30` |

## Debugging

If data is not appearing in Kibana:
1. **Check Bridge Logs:**
   ```bash
   # Compose
   docker compose logs -f parquet-es-bridge
   
   # Kubernetes
   kubectl logs -n weather deploy/parquet-es-bridge -f
   ```
2. **Verify Parquet Files Exist:** Ensure `central-station` is actually flushing files to the disk (controlled by `ARCHIVER_FLUSH_INTERVAL_MS`).
3. **Check ES Connection:** The bridge logs will report `Elasticsearch connection successful` on startup. If not, verify the `ES_HOST` URL.
